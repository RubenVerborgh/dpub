EPUB 3

EPUB 3 is a very mature specification

  • Widely used format for digital publication
  • The only truly portable format (and not bound to a particular environment)
  • Can be used for books, magazines, journal articles, documentation…

What is the secret?

Well, at least one of the secrets…

  • EPUB is based on Web Technologies like HTML, CSS, SVG, etc.
    • collectively referred to as “Open Web Platform” (OWP)
  • I.e., no rendering, formatting, etc, specification had to be defined
  • Instead, EPUB relies on specifications defined by W3C and deployed, e.g., in browsers

Are we all done? Ie, is OWP perfect for EPUB?

There are two major areas that need work:

  1. bring OWP and Digital Publishing even closer
  2. bring the Web and Digital Publishing even closer

Bring OWP and Digital Publishing even closer

Missing OWP features

  • Some features, needed by publishing, are missing from OWP; examples:
    • pagination, page controls for headings, page numbers
    • graphics transition effects (needed in, e.g., mangas, cartoons, children books)
    • accessibility oriented structuring of complex documents
    • dynamic numbering of pages, index references, etc, over several (HTML) files
  • W3C has a Digital Publishing Interest Group to look into these issues and cooperate with the relevant W3C groups

Bring OWP features to publishing

  • For example:
    • dynamic tools deployed on the Web would be beneficial in publishing
    • educational publications should use external services on the Web via extra protocols
    • new authentication methods could change the way publications are accessed
    • etc.

Bring the Web and Digital Publishing even closer

What is, in fact, a (digital) book?

A book is a discrete collection of text (and other media), that is designed by an author(s) as an internally complete representation of an idea, or set of ideas; emotion or set of emotions; and transmitted to readers in various formats.

Hugh McGuire: “What books can learn from the Web / What the Web can learn from books”, Medium

  • This is independent of format: the same “book” should be available in different format and media
  • Instead: in practice, a Digital Book is different than a Web page (conceptually and practically)
  • This should not be the case!

What does this mean?

Separation between publishing “online”, as Web sites, and offline and/or packaged should be diminished to zero

  • This means:
    • publication content on the Web can be loaded into a browser or a specialized reader, whatever the user prefers
    • a publication on a local disc can be pushed onto the Web and used without any change
    • content are authored regardless of where they are used
    • these are done without any user interaction, possibly automatically

What does this mean?

ibta arabia

For example: book in a browser

  • On a desktop I may want to read a book just like a Web page:
    • easily follow a link “out” of the book
    • create bookmarks to “within” a page in a book
    • use useful plugins and tools that my browser may have
    • create annotations
    • sometimes I may need the computing power of my desk-top for, e.g., interactive 3D content

For example: book in a browser

  • But, at other times, I may also want to use a small dedicated reader device to read the book on the beach…
  • All these on the same book (not conversions from one format to the other)!

For example: I may not be online…

  • I may find an article on the Web that I want to review, annotate, etc., while commuting home on a train
  • I want the results of the annotations to be back online, when I am back on the Internet
    • note: some browsers have an “archiving” possibility, but they are not interoperable

For example: scholarly publishing

  • Paper may be published on-line, but people may want to download it for offline use
  • The format of the paper should be adaptable to reading environment
    • no two column, fixed layout file that cannot be handled on an my iPad…
  • The “paper” may also contain video, audio, data, programs…
    • scholarly publishing is not text only any more!

For example: in-house publishing

  • Major companies (IBM, Intel, Boeing, FAO, Renault,…) are specialized “publishers” publishing a huge amount of documentation
  • Delivering it on paper is not an option any more
  • Fast refresh time is needed
  • The same document should be available offline (e.g., in the cockpit) or online (e.g., on the work floor): there should be no difference between the two

For example: educational publications

  • What is an educational publication?
    • a book that requires offline access?
    • a packaged application with built-in interactive tests, animated examples?
    • a Web client reaching out to Web services for assessing test results, to encyclopedia, …?
    • an interactive data container storing various data for, e.g., demonstrations?
  • The borderline between a “book” and a “(Web) Application” is becoming blurred…

But… why not rely only on the Web?
(i.e., forget about downloaded content, it is outdated!)

Several reasons…

  • The future may be that everyone is always connected… but the reality is different for many years to come
    • slow connections, e.g., or on a plane or bus or some geographical areas
    • huge roaming prices among countries
  • As a consequence, offline access is important:
    • students, pilots in a cockpit, doctors, etc., must have a secure access to documents under all circumstances
    • schools, governments, etc., may not update equipments frequently; it is necessary for publications to be accessible on current as well as older browsers, software, etc.
    • readers, students, scholars, etc., may need to have access to their books also when abroad

It is not only a matter of access possibilities

  • A publication may consist of many resources (text, images, videos, etc), but the abstract “work” should be available as a single, conceptual unit
  • There are a number of use cases for that:
    • the whole book should be archived in a national library
    • the publication has to be checked for integrity as a whole (e.g., a legal publication)
    • sharing a publication with a friend requires the concept of the publication as a whole (whether offline or online)
    • publishing business models are based on the concept of “works”
  • A simple Web address to a single page is not enough!

There are also technical reasons for the “single unit” concept

  • Value counters (page counters, section numbering, footnotes, endnotes, etc.) should be consistent across all constituents of a publication
  • User preferences should apply to the “work” rather to a single Web page
  • Any search facility for a book should have access to the whole book

Synergy effects of convergence

Advantage for the publishers‘ community

    • The main interest of publishers is to produce, edit, curate, etc., content
    • Publishers have invested heavily into technology developments, but the Web developers’ community can complement that with a wider reach and perspective
    • Working closely with Web developers avoids re-inventing wheels

Advantage for publishers‘ community

  • OWP also defines API-s to access to, e.g., system resources or utilities
    • index database, Web storage, battery status API, real-time communication, geolocation,…
  • Aligning more on OWP means that specialized readers or Web sites, software, services, etc, can rely on those

Advantage for the Web community

  • Publishers have experience in:
    • ergonomics, typography, aesthetics…
    • publishing long texts, with the right readability and structure
  • Workflow for producing complex content

Synergy effects in general
